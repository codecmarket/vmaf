# VMAF CUDA Baseline Benchmark — 2026-04-19

Baseline measurement of the current CUDA path on two GPUs, before any
modernization work. Captured to identify where per-frame time is spent
and set a reference point for subsequent optimization.

## Setup

- **Build:** libvmaf at `966be8d5` (master), release, `enable_cuda=true`, `enable_nvcc=true`.
- **Toolchain:** CUDA 13.2.78, gcc 14.2.0, Debian 13, Linux 6.19.11.
- **GPUs:** device 0 = NVIDIA RTX 5070 (Blackwell, compute 12.0); device 1 = NVIDIA RTX 4070 (Ada, compute 8.9). Selected per run via `CUDA_VISIBLE_DEVICES`.
- **Clips:** Xiph derf `crowd_run`, first 300 frames. Distorted variants produced by Vulkan-H.264 encode (ffmpeg 8.1 `h264_vulkan`) at deliberately low bitrates (300k / 1M / 4M for 480p / 1080p / 2160p) so VMAF scores fall away from saturation. 480p derived from 1080p via Lanczos downscale.
- **Command per run:** `nsys profile --trace=cuda,nvtx,osrt -- vmaf --gpumask 0 --model version=vmaf_v0.6.1 --frame_cnt 300 ...`
- **Model:** default `vmaf_v0.6.1` (ADM + VIF + motion).

## Key findings

**1. The newer, faster GPU is slower.** RTX 5070 loses to RTX 4070 by 6–16% at every resolution. The 5070 is a Blackwell card with more SMs and higher memory bandwidth than the Ada 4070. If the pipeline were GPU-bound, the 5070 would win. It doesn't — so the pipeline is **not** GPU-bound.

**2. Throughput barely scales with resolution.** 1080p is only ~15% slower than 480p despite 4.5× more pixels. A pixel-throughput-limited pipeline would be roughly 4× slower. The plateau at small frames means something other than per-pixel work is dominating.

**3. Per-frame kernel launch count is constant (42) across all three resolutions.** Confirmed via `nsys stats --report cuda_api_sum`: `cuLaunchKernel` called exactly 12,600 times per 300-frame run regardless of clip size. Per-frame launch count is fixed by the feature-extraction DAG.

**4. The GPU is mostly idle.** Summing active kernel time from the nsys traces:

| GPU       | Resolution | Wall/frame | Kernel active/frame | GPU idle/frame |
|-----------|-----------:|-----------:|--------------------:|---------------:|
| RTX 5070  |       480p |   17.3 ms  |             ~0.4 ms |       ~16.9 ms (97.4% idle) |
| RTX 5070  |      1080p |   19.5 ms  |             ~1.5 ms |       ~18.0 ms (92.3% idle) |
| RTX 5070  |      2160p |   30.0 ms  |             ~7.8 ms |       ~22.2 ms (74.0% idle) |

**5. Per-frame syncs dominate the timeline.** `cuStreamSynchronize` fires 2,411 times per 300-frame run (≈8 per frame), consistent with the synchronize-then-extract pattern in each feature: one sync at the top of `extract_fex_cuda` plus one mid-extract before the D2H copy, repeated across ADM/VIF/motion. At 480p this is where the ~17 ms idle gap lives.

## Interpretation

This is a CPU-serialized pipeline. Each feature's `extract_fex_cuda` begins
with `cuStreamSynchronize(s->str)` to wait for the previous frame's
`cuLaunchHostFunc` callback to finish consuming the single-buffered host
result slots (`accum_host`, `sad_host`, `cpu_param_buf`). Until those
slots free, frame N+1 cannot touch them. Result: GPU work for frame N+1
cannot overlap with host callback work for frame N.

At small resolutions where GPU kernels finish in sub-millisecond, the
sync is the whole frame. At 2160p, kernels take enough that the sync
matters less, but even at 2160p ~74% of GPU time is idle.

## Projected headroom

With per-feature host state K-buffered (`K >= max in-flight frames`,
typical 2–4) and the two `cuStreamSynchronize` calls per extract
removed, the per-frame lower bound should approach max(GPU active time,
CPU callback time). Naive projection:

- **480p:** current 17.3 ms → plausible floor ~2–3 ms (≈ 300–500 fps).
- **1080p:** current 19.5 ms → plausible floor ~4–5 ms (≈ 200+ fps).
- **2160p:** current 30 ms → plausible floor ~10 ms (≈ 100 fps).

These are upper-bound projections assuming the serialization is the
*only* remaining cost. Real-world result will depend on how well the
CPU callback thread keeps up; that is measurable from the re-run
traces.

## Artifacts in this directory

- `summary.csv` — machine-readable wall-clock + score per cell.
- `<gpu>_<res>.log` — stdout/stderr per run.
- `<gpu>_<res>.vmaf.json` — VMAF JSON output per run.
- `<gpu>_<res>.nsys-rep` — nsys binary trace (open with `nsys-ui` or
  `nsys stats`).
- `<gpu>_<res>.sqlite` — auto-generated on first `nsys stats` invocation
  (large; can be deleted).
