# VMAF CUDA Lever-1 K-Buffering — 2026-04-19

Removes the per-extract `cuStreamSynchronize(s->str)` calls from the three
CUDA feature extractors (ADM, VIF, motion) by K-buffering (K=4) the
per-slot host state (`accum_host` / `sad_host` / `results_host`,
`write_score_parameters_*`, plus a CUevent per slot). Frame N+K's extract
blocks only on its own slot's `consumed` event, not on frame N+1-K's.

## What changed

Each extractor now keeps a ring of `K=4` `*Slot` structs containing the
host pinned buffer, `write_score_parameters_*`, and a `consumed` event.
Per extract:

1. `cuEventSynchronize(slot->consumed)` — blocks only if all K slots in
   flight.
2. Queue GPU work writing into `slot->*_host` via
   `cuMemcpyDtoHAsync(..., s->str)`.
3. `cuLaunchHostFunc(..., write_scores, &slot->cpu_param)` — callback
   reads the slot's own host buffer, not the shared `s->buf` legacy
   pointer.
4. `cuEventRecord(slot->consumed, s->str)` — slot can be reused K frames
   later.

Shared device state (`accum_data` in VIF, `sad` in motion, `tmp_res` +
dwt2 bands in ADM) is still single-buffered; the old top-of-extract
`cuStreamSynchronize(s->str)` was doing double duty by also serializing
access to those device buffers across frames. Its removal surfaced a
race between:

- VIF scale-0 `filter1d` on the picture stream writing `accum_data`,
  versus the prior frame's D2H of `accum_data` on `s->str`.
- ADM scale-0 `dwt2` on the picture stream writing `ref_dwt2`/`dis_dwt2`
  versus the prior frame's downstream consumption on `s->str`.

Fixed by recording `s->finished` on `s->str` (after memset / already
at end of extract for ADM) and adding `cuStreamWaitEvent(picture_stream,
s->finished, …)` before the scale-0 kernels fire.

## Why no speedup

Baseline profile (see `baseline-2026-04-19.md`) showed 97% GPU idle at
480p with 2411 `cuStreamSynchronize` calls. Post-change counts:

| API                    | Baseline | K-buffer |
|------------------------|---------:|---------:|
| `cuStreamSynchronize`  |     2411 |      614 |
| `cuEventSynchronize`   |      600 |     1500 |
| `cuStreamWaitEvent`    |     2699 |     3299 |
| `cuEventRecord`        |     3303 |     4202 |

We swapped ~1800 host-side stream drains for ~900 event waits (each
<2 µs). The host serialization point moved, but the **GPU-side
serialization didn't change**: the new `cuStreamWaitEvent(picture_stream,
s->finished)` fences serialize frame N+1's scale-0 kernels on the prior
frame's D2H, because `accum_data` / dwt2 bands are shared across frames.

Effectively, one necessary wait was replaced with another. Correctness
is preserved, frames 2..K can be queued without the host blocking, but
the GPU timeline still processes one frame's feature-extract work at a
time — so wall-clock is unchanged.

## What would actually unlock throughput

Lever 2: K-buffer the device state too. Specifically:

- `VifBufferCuda::accum_data` → `accum_data[K]`, selected per slot
- `AdmBufferCuda::tmp_res` → `tmp_res[K]`
- `AdmBufferCuda::ref_dwt2`, `dis_dwt2`, `i4_ref_dwt2`, `i4_dis_dwt2`,
  `tmp_ref`, `tmp_dis`, `decouple_r`, `decouple_a`, `csf_a`, `csf_f`,
  `adm_cm[*]`, `adm_csf_den[*]` → all K-deep
- Motion's `sad` is 8 bytes — cheap to K-buffer

This would let the picture-stream fences drop and K frames could run
in parallel on the GPU. Risk: the dwt2-band set is large at 2160p
(~tens of MB per slot), so memory footprint grows ~K×.

A simpler alternative: run features on separate streams per frame (per
slot). Needs deeper rework of the extractor state.
