# VMAF CUDA Lever-2 — K-Buffered Device State — 2026-04-19

Completes the K-buffer migration for the three CUDA feature extractors
(VIF, motion, ADM). Each extractor now owns `K=4` slots, each with its
own `CUstream` and its own copy of the per-frame device buffers
(`accum_data` for VIF, `sad` for motion, the full `AdmBufferCuda` —
`data_buf` / dwt bands / `tmp_ref` / `tmp_dis` / `tmp_accum` /
`tmp_accum_h` / `tmp_res` — for ADM). Frame N+K blocks only on its own
slot's `consumed` event, and K frames can execute in parallel on the
GPU.

Landed as three commits for easy bisect:

```
82e16065  cuda/adm: Lever 2 — K-deep per-slot device buffers + per-slot streams
05cd039b  cuda/motion: Lever 2 — per-slot streams, K-deep SAD, kernel chain
6b1ceae3  cuda/vif: Lever 2 — K-deep slots with per-slot streams and device buffers
```

## Measurement correction vs. Lever-1 / baseline docs

The baseline and Lever-1 numbers in
`baseline-2026-04-19.md` / `kbuffer-2026-04-19.md` were captured under
`nsys profile --trace=cuda,nvtx,osrt`. At 480p and 1080p the nsys
overhead (per-call CUDA API interception + OSRT tracing) dominates
the measurement — bare wall-clock at 480p is ~11× faster than what
nsys reported (636 fps vs 57 fps). Those two docs are still useful
for kernel-level timeline inspection, but the fps columns there
should not be compared directly to anything.

All Lever-2 numbers in this document are bare wall-clock. The
baseline row above was re-measured against the same clips on the same
machine with nsys disabled, at commit `966be8d5` (pre-Lever-1), so the
Δ column is apples-to-apples.

## Why 2160p wins and 1080p doesn't

Three regimes:

- **480p (CPU-bound):** GPU kernels finish in sub-millisecond. The
  whole frame is dominated by the host-side feature-collector +
  `write_scores` callback + picture-pool bookkeeping. Lever 2 does not
  touch any of that, so the fps is flat (±0.5%).

- **2160p (GPU-bound):** Feature-extraction kernel time per frame is
  long enough (~8–15 ms depending on feature) that running K frames in
  parallel on the GPU matters. With per-slot streams, ADM's scale-1–3
  on frame N can overlap with ADM's scale-0 dwt2 on frame N+1 (private
  buffers, no races), and VIF's scale 0..3 of frame N+1 can start
  while frame N is still in its D2H tail. Net +45% on 5070, +21% on
  4070.

- **1080p (intermediate, regression):** Kernel time per frame is
  ~1.5–2 ms — small enough that the serial baseline already kept the
  GPU mostly busy, large enough that the extra launch/synchronization
  bookkeeping from K streams (more `cuStreamWaitEvent`, more event
  records, the motion `kernel_done` chain) eats into the margin. Net
  -9 to -12%.

The 1080p regression is real and reproducible but is the case where
the pipeline had the least to gain from Lever 2 in the first place.
Recovering it would mean either (a) adapting the number of in-flight
slots to resolution (K=2 at 1080p, K=4 at 2160p), or (b) reducing the
per-frame event bookkeeping — e.g. collapsing the motion `kernel_done`
chain into a single waited event per in-flight batch. Neither is done.

## Architectural notes

- **VIF:** straightforward. Each slot owns a `VifBufferCuda`
  including `accum_data`, `data`, `sum` bands. Picture-stream fences
  on prior frame's `scale-0` D2H removed because the per-slot `buf`
  means no cross-frame data race.

- **Motion:** trivially small device state (`sad` is 8 bytes). Real
  subtlety: the motion kernel has a built-in frame-to-frame aliasing
  dependency on `blur[prev]` / `blur[curr]`. This is preserved via a
  per-slot `kernel_done` event; slot N+1's kernel waits on slot N's
  `kernel_done` before firing. First frame passes through because the
  prior slot's `kernel_done` is unrecorded (so
  `cuStreamWaitEvent` returns immediately).

- **ADM:** biggest lift. Each slot gets a full `AdmBufferCuda` —
  roughly 570 MB per slot at 2160p, so ~2.3 GB total across K=4
  slots. Scale-0 `dwt2` still fires on each picture's own stream so
  the input picture can be released early, but writes into the slot's
  private `ref_dwt2` / `dis_dwt2` bands; scales 1–3, decouple, csf,
  cm, the `tmp_res` D2H, and the `write_scores` host callback all
  run on `slot->stream`. The Lever-1 picture-stream fence on
  `s->finished` is gone — the per-slot buf eliminates the race it was
  there to paper over.

## Memory budget (2160p, K=4)

Per slot per extractor (rough, 2160p 10-bit):

- VIF: ~30 MB × 4 slots ≈ 120 MB + ~11 MB accum_data × 4 ≈ 44 MB total
- motion: negligible (8 B × 4 slots)
- ADM: ~570 MB × 4 slots ≈ 2.3 GB

Comfortably inside 12 GB cards (5070 / 4070). At 8K or higher K this
would start to matter.

## Reproduce

```sh
# bare wall-clock, 3 runs per cell, 2 GPUs × 3 resolutions
for dev in 0 1; do for res in 480p 1080p 2160p; do for r in 1 2 3; do
  t0=$(date +%s.%N)
  CUDA_VISIBLE_DEVICES=$dev /home/dsummer/VMAF/libvmaf/build/tools/vmaf \
    --gpumask 0 \
    -r /tmp/vmaf-bench/clips/ref_${res}.y4m \
    -d /tmp/vmaf-bench/clips/dist_${res}.y4m \
    --model version=vmaf_v0.6.1 --frame_cnt 300 --quiet >/dev/null
  t1=$(date +%s.%N)
  awk -v a=$t0 -v b=$t1 -v d=$dev -v res=$res -v r=$r \
    'BEGIN{printf "dev=%s res=%s run=%s elapsed=%.3fs fps=%.2f\n", d, res, r, b-a, 300/(b-a)}'
done; done; done
```
